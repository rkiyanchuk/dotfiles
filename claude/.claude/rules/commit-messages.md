# Commit messages

. Type
prefixes are redundant (the description already implies the type), restrictive
(many changes are bug/refactor/feature at once), and they bury the one thing
readers actually care about: the *scope* — the area of the codebase touched.

Use the **scope-prefixed** style favored by Linux, Git, Go, FreeBSD, and
nixpkgs:

```
scope: short description in the imperative mood
```

The scope is whatever is natural for the project: subsystem, package path,
microservice, module, or directory. Examples:

```
claude: update plugin install list
git: fix recursive-sync depth limit
nvim: add blink.cmp signature help
```

When a change is genuinely cross-cutting and no single scope fits, a bare
description is acceptable — but prefer a scope whenever one is self-evident.

## Rules

- Do **not** use Conventional Commits (`feat:`, `fix:`, `chore:`, etc.)
- **Subject line ≤ 50 characters.** Make the title informative.
- The subject is followed by **one blank line**, then the body (if any).

### Subject line

- Imperative mood: "add", "fix", "remove" — not "added"/"adds"/"fixing".
  It should complete the sentence "If applied, this commit will _____".
- Capitalize the description's first word (after the `scope:` prefix).
- No trailing period.

## Body

- Optional, but include it for any non-trivial change.
- Wrap at **72 characters**.
- Explain **what** changed and **why** — the motivation and context.
  Leave the how* to the diff itself.
- Use blank lines to separate paragraphs; bullet lists are fine.

## Trailers

Structured trailers go at the end, after a blank line, as `Key: value` lines
(RFC-822 style; parseable by `git interpret-trailers`). Use the standard,
widely-recognized keys when applicable:

- `Signed-off-by: Name <email>` — Developer Certificate of Origin (`git commit -s`).
- `Co-authored-by: Name <email>` — additional human authors.
- `Reviewed-by: Name <email>` — who reviewed the change.
- `Tested-by: Name <email>` — who verified it.
- `Reported-by: Name <email>` — who reported the issue.
- `Fixes: <issue/commit>` / `Closes: #123` — reference the issue or commit.
- `Link: <url>` — related discussion, ticket, or docs.

### Claude Code attribution

Attribution depends on who authored the **code**, not who wrote the message:

- When the **code being committed was generated by Claude Code**, add a
  co-author trailer:

  ```gitcommit
  Co-authored-by: Claude Code <noreply@anthropic.com>
  ```

- When Claude Code only **wrote the commit message** but a human wrote the
  code, do **not** add any Claude attribution.
